# Retrieval and Action System

Knowledge bases are useful here only when they help later work happen. The goal is not to store every thought. The goal is to make future retrieval and action easier.

## Reusable Pattern

A useful knowledge-work system has this shape:

- Keep hot routing short.
- Keep canonical project meaning in a small number of clear sources.
- Keep durable research memory in a knowledge base.
- Separate confirmed facts, user corrections, hypotheses, source notes, tool limitations, TODOs, and observed lessons.
- Preserve uncertainty explicitly instead of turning open questions into requirements.
- Use backlinks and indexes so colder notes remain discoverable without loading everything.
- Keep models, metrics, summaries, tests, and dashboards answerable to evidence or action so they do not become self-certifying artifacts.

## Practical Rule

Every durable note should help with at least one later action:

- start a project;
- inspect current state;
- decide between alternatives;
- revise a draft or spec;
- test an assumption;
- notice a recurring pattern.

If a note does not support retrieval or action, leave it as rough capture until its role becomes clear.
